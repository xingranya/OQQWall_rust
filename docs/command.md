# OQQWall_RUST 指令手册（docs/command.md）

> 本文档用于开发/运维/管理员使用与对齐。  
> 设计目标：**尽可能兼容原版 OQQWall 的群内指令输入方式与语义**，在 Rust 重构版中保留相同的“全局指令 / 审核指令”体验；实现细节与数据结构不要求兼容原版。  
> 参考来源：原版 `getmsgserv/command.sh` 的帮助文本与指令分派逻辑（全局指令与审核指令均在脚本中定义）。

---
## 0. 审核请求发送

Rust 版沿用原版“在审核群推送摘要 + 渲染 + 原图”的体验：

- 摘要文本：`#<review_code> 来自 <昵称>(<QQ>)`，并附“消息概览”逐条列出文本，多媒体以 `[图片]/[视频]/[文件]/[合并转发]` 形式提示，合并转发会展开内容（review_code 为审核内部编号，仅审核侧可见，不对外展示）。
- 渲染模式：直接生成 PNG，审核群收到“摘要 + 渲染图 + 原图列表”。
- 附件：投稿里的原始图片/表情包会紧跟摘要/渲染图发出便于放大；视频/文件保持卡片或占位显示。
- 在 QQ 中通常是一串消息，例如：
```
#123 来自昵称(qq号)
消息概览：
 文本xxxxx
 [图片]
[合并转发消息]——
xxxxx
xxxxx
[图片]
xxxxx
——————
文本xxxxxx
图片：
[原图1]
[原图2]
```

## 1. 指令触发方式

原版 OQQWall 支持两种常见触发方式：

1) **@机器人执行（推荐兼容）**  
- 全局指令语法：`@本账号/次要账号 指令`  
- 审核指令语法：`@本账号 review_code 指令 [参数...]`  

2) **回复审核消息执行（推荐兼容）**  
- 审核指令也可“回复审核消息，只发指令”  

> Rust 版建议：两种都支持，并优先“回复审核消息”以减少输入内部编号的错误。

## 2. 核心概念

### 2.1 审核编号（review_code）
原版脚本把“审核编号”作为第一参数 `object`，并将其当作数字分支处理：`case $object in [0-9]*) ...`。  
同时要求对应目录存在（如 `./cache/prepost/$object`）才可执行。

> Rust 版建议：review_code 为系统内部给每个稿件分配的唯一编号，仅用于审核/运维，不对外展示；命令输入仍需保持原版习惯即可。

### 2.2 指令分词（兼容性要点）
原版脚本用 `awk` 取前三段：`object`/`command`/`flag`。  
这意味着：**原版天然对“多空格、多词参数”支持较弱**（例如 `评论`/`回复` 的内容在原实现里更像单 token）。

> Rust 版建议（兼容+增强）：  
> - 仍然兼容 “前三段” 语义：`object command flag`  
> - 同时支持 “尾部参数整段”：将第 3 段及之后拼成 `args_text`（支持空格与引用）  
> - 解析优先级：  
>   1) 若是 `快捷回复 添加 指令名=内容`，优先按 `=` 切分  
>   2) 其它命令按 `object command [rest...]` 解析

### 2.3 外部编号（numfinal）
- 原版为每个账号组维护 `cache/numb/<group>_numfinal.txt` 作为“外部编号/可见编号”，用于最终发布到 QQ 空间时对外展示；审核阶段仅展示 review_code。
- 初次部署必须通过“设定编号”初始化；`是/立即` 在写入发送调度后会自增编号，`否` 也会自增并标记该稿件已处理（由人工手动进入 QQ 发送）；“删除暂存区”会把编号回滚到暂存区中最小的 review_code（撤销尚未发送的递增）。
- Rust 版若保留外部编号，需保证编号递增/回滚逻辑与账号组绑定且幂等。

### 2.4 暂存区（sendstorge）
- 审核通过的投稿会先进入组级暂存区（SQLite 表 `sendstorge_<group>` + `cache/prepost/<review_code>`），由 sendcontrol 按规则批量发送。
- 触发发送：达到 `max_post_stack`、图片总数超过 `max_image_number_one_post`、命中 `send_schedule` 时间点，或管理员执行“发送暂存区”。
- “删除暂存区”会删除暂存稿件及渲染产物，并把外部编号回滚到暂存区最小 review_code；不会把稿件退回待审核，如需找回请用“调出”。
- 若 `max_post_stack == 1`（原版默认），暂存区等价于关闭：每稿单独发送，网页审核也会隐藏暂存区面板。

## 3. 全局指令（任何时刻可用）

原版帮助文本声明：这些是“任何时刻@本账号调用的指令”，语法为 `@本账号/次要账号 指令`。

> 下表为 Rust 版建议“保持同名同义”，并在实现时对应到事件系统（例如 `OutboundReplyRequested`、`SystemCheckRequested` 等）。

| 指令 | 语法 | 原版语义/说明 | Rust 版实现建议 | 可用性 |
|---|---|---|---|---|
| 帮助 | `@机器人 帮助` | 输出完整帮助列表（全局+审核指令） | 直接回复 `command.md` 的精简版 + 链接到完整文档 | ✅ |
| 调出 | `@机器人 调出 <review_code>` | 调出曾经接收到过的投稿，执行 `preprocess.sh <review_code> renderonly` | 映射为：根据 review_code/post_id 重新生成渲染产物并展示/发送预览 | ✅ |
| 信息 | `@机器人 信息 <review_code>` | 查询该编号的接收者、发送者、所属组、处理后 JSON 等 | 映射为：输出 post 元信息（来源、账号组、状态、失败原因、产物 id） | ❌ |
| 手动重新登录 | `@机器人 手动重新登录` | 扫码登录 QQ 空间 | 触发 daemon/driver 进入“需要人工扫码”流程并提示操作 | ❌ |
| 自动重新登录 | `@机器人 自动重新登录` | 尝试自动登录 QQ 空间 | 触发自动刷新会话；失败则进入人工介入态 | ❌ |
| 待处理 | `@机器人 待处理` | 列出当前等待处理投稿（按账号组过滤） | 输出待审核/待发送列表（按账号组） | ✅ |
| 删除待处理 | `@机器人 删除待处理` | 清空待处理列表；原注释称“相当于对列表中的所有项目执行‘删’审核指令” | 需要谨慎：建议仅管理员可用；实现为批量归档/删除待审核项 | ✅ |
| 删除暂存区 | `@机器人 删除暂存区` | 清空暂存区内容，并回滚外部编号 | Rust 版若无“暂存区”概念，可映射为：清空发送队列（并记录审计日志） | ✅ |
| 发送暂存区 | `@机器人 发送暂存区` | 将暂存区内容发送到 QQ 空间（通过 sendcontrol flush） | 映射为：立即触发 scheduler/sender flush（忽略 not_before，按顺序发送） | ✅ |
| 列出拉黑 | `@机器人 列出拉黑` | 列出当前被拉黑账号列表 | 输出黑名单（sender_id -> reason） | ❌ |
| 取消拉黑 | `@机器人 取消拉黑 <senderid>` | 取消对某账号拉黑 | 黑名单删除并确认 | ❌ |
| 设定编号 | `@机器人 设定编号 <纯数字>` | 设定下一条说说外部编号 | 若 Rust 版有“外部编号”，保留；否则可做为“显示编号前缀”配置项 | ❌ |
| 快捷回复 | `@机器人 快捷回复` | 列出当前账号组快捷回复列表 | 输出模板列表 | ✅ |
| 快捷回复 添加 | `@机器人 快捷回复 添加 指令名=内容` | 添加快捷回复；且会检查不与审核指令冲突 | 实时更新当前组并写回配置文件 | ✅ |
| 快捷回复 删除 | `@机器人 快捷回复 删除 指令名` | 删除快捷回复 | 实时删除当前组模板并写回配置文件 | ✅ |
| 自检 | `@机器人 自检` | 系统自检（CPU/内存/硬盘/服务状态），必要时尝试重启服务 | Rust 版输出 health summary；重启动作通过 daemon 实现 | ❌ |
| 系统修复 | `@机器人 系统修复` | 重启除 serv.py 外服务并重建 UDS（强修复） | Rust 版：重启 drivers/清理 socket| ❌ |

## 4. 审核指令（只在审核流程中使用）

原版帮助明确：审核指令语法为 `@本账号 review_code 指令` 或 “回复审核消息 指令”。  
审核指令清单与说明如下（原版描述）：

> Rust 版注意：你当前阶段不做 AI，所以部分“刷新/重渲染/消息全选/匿”等会变成“纯渲染/纯重跑流水线”的含义。为了兼容体验，仍建议保留同名命令。

| 指令 | 语法（两种） | 原版语义/说明 | Rust 版建议实现 | 可用性 |
|---|---|---|---|---|
| 是 | `@机器人 <review_code> 是` 或 回复 `是` | 发送，并给稿件发送者发送成功提示 | 等价于“通过并入队发送”；成功后可私聊通知投稿人 | ✅ |
| 否 | `… 否` | 机器跳过此条，人工处理（常用于分段/匿名失败或含视频） | 外部编号+1，标记稿件已处理并进入“人工发送”状态；管理员需手动进入 QQ 发送 | ✅ |
| 匿 | `… 匿` | 切换匿名状态，处理后会再次询问指令 | 当前无 AI：可改为“匿名开关”字段切换 + 重新渲染并重新发布审核 | ✅ |
| 等 | `… 等` | 等待 180 秒后重新执行分段-渲染-审核 | 映射为：`ReviewDelayed(now+180s)` + 到点重跑渲染/发布 | ✅ |
| 删 | `… 删` | 此条不发送，也不用人工发送；外部编号+1 | 映射为：归档/删除草稿（是否+1 外部编号视 Rust 版实现） | ✅ |
| 拒 | `… 拒` | 拒绝稿件；给发送者发送被拒提示 | 映射为：Rejected + 私聊通知 | ✅ |
| 立即 | `… 立即` | 立刻发送暂存区全部投稿，并立即把当前投稿单发 | 映射为：队列 flush + 当前 post 优先级提升到最高并立刻发送 | ✅ |
| 刷新 | `… 刷新` | 重新进行“聊天记录->图片” | 无 AI：重跑渲染（从原始 ingress 重建 draft → 渲染） | ✅ |
| 重渲染 | `… 重渲染` | 重做渲染但不重做 AI 分段（调试渲染） | 直接重新渲染（draft 不变） | ✅ |
| 消息全选 | `… 消息全选` | 强制把本次投稿所有消息作为内容并重渲染 | 映射为：draft builder 使用 “all messages” 模式重建 blocks | ✅ |
| 合并 | `@机器人 <review_code_a> 合并 <review_code_b>` 或 回复其中一条 `合并 <review_code>` | 合并两个来自同一投稿人的稿件 | 校验同一投稿人/同一组，合并 ingress 顺序后重建 draft → 重新发布审核 | ✅ |
| 扩列审查 | `… 扩列审查` | 扩列审核流程（抓等级/空间/名片/二维码等） | Rust 版可先保留命令但返回“未实现/已禁用”，或做插件化 | ❌ |
| 评论 | `… 评论 <文本>` | 增加文本评论，处理后再次询问 | 映射为：给 post 增加评论 block → 重新渲染 → 重新发布审核 | ❌ |
| 回复 | `… 回复 <文本>` | 向投稿人发送一条信息 | OneBot 私聊投稿人 | ✅ |
| 展示 | `… 展示` | 展示稿件内容 | 回复渲染链接/输出文本摘要 | ❌ |
| 拉黑 | `… 拉黑 [理由]` | 不再接收来自此人的投稿 | 将 sender_id 加入黑名单（可带理由） | ✅ |
| 快捷回复指令 | `… <快捷指令名>` | 使用预设模板向投稿人发送消息 | 根据账号组 quick_replies 模板渲染并私聊发送 | ✅ |

## 5. 权限与作用域

原版脚本存在“按账号组权限判断”的逻辑（例如执行对象 review_code 时会检查其所属 ACgroup 是否等于当前 groupname，否者报权限错误）。  

明确两条权限规则：

1) **全局指令**：仅允许审核群内管理员执行  
2) **审核指令**：仅允许审核群内管理员执行，并且限定只能操作本账号组的 post（匹配 group）
- 组判定来自配置的 `accounts -> groupname` 映射（`accounts[0]` 为主账号）；多账号场景下只能操作当前机器人所属组的投稿。
- 审核相关群消息（审核推送/审核指令响应）仅由当前有效主账号发送；当主账号离线时按 `accounts` 顺序自动替补。
- 管理员判定沿用 QQ 群权限（群主/管理员），而非额外名单；web 审核登录账号也需要在群内具备管理员权限才可执行指令。

## 6. 兼容性与增强建议（给开发用）

### 6.1 兼容原版两种输入形式
- 完整形式：`@机器人 <review_code> <指令> [参数...]`  
- 回复形式：回复审核消息，仅输入 `<指令> [参数...]`  

### 6.2 Rust 版建议的解析规范（兼容+更好用）
- 先判断是否“回复审核消息”：若是，默认绑定到该审核项（无需内部编号）
- 若不是回复：支持两种格式  
  1) `<review_code> <指令> [args...]`  
  2) `<指令> [args...]`（仅对全局指令）
- 参数规则：`args_text` 是剩余所有内容（支持空格），避免原版只取前三段的局限

## 8. 实现对照（开发备注）

- 原版帮助文本包含“全局指令”和“审核指令”的完整列表与语义描述（见原版 `getmsgserv/command.sh`）。  
- 原版对输入拆分 `object/command/flag` 的方式来自脚本的 `awk` 分词。  
- 数字 object（review_code）分支会进入针对审核项的处理流程，并且有组权限检查。
- Rust 版建议：review_code 即审核内部编号；外部编号独立，只用于最终对外展示的 Qzone 编号。
